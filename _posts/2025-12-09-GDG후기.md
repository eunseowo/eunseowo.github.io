### [업스테이지] AI로 굴러가게 만들기: 작은 팀의 생존

첫번째로 참여한 세션이다. 개인적으로 기술에 관한 것보다 AI를 실무에 적용한 사례를 알아보고 싶은 생각이 들었었는데, 이런 세션이 있다길래 바로 들어갔다.

특히 인상 깊었던 점은 AI 시대에 우리가 가져야 할 세 가지 변화에 대한 이야기였다.

**1. 먼저 만들고, 나중에 정리한다.**
일단 만들어서 보여주고, 피드백을 듣고, 그걸 기준으로 다시 만든다. 즉 완벽함과 계획주의를 버리라는 것이다. 

**2. 역할을 나누지 않고, 일을 나눈다.**
요즘은 디자이너도 코드를 수정하는 시대이다. 기획자가 문서 대신 코드를 작성하는 일도 많다고 하니... 이걸 누가 해야 하나?가 아니라 이걸 누가 제일 빨리 할 수 있지?로 판단하는 역량을 키워야될 것 같다. 

**3. 나를 AI로 확장시킨다 - 나의 팀원**
AI의 가장 큰 장점은 온보딩이 필요 없다는 것이다. 전문성을 즉시 빌려올 수 있는 팀원이 생기는 셈이다. 예를 들어, 코드 리뷰를 할 때 AI가 구조, 네이밍, 컨벤션 같은 기본적인 지적을 1차로 수행하게 한다면, 개발자는 오롯이 설계의 품질과 로직의 본질에만 집중할 수 있다.

![251209_4.png](/assets/img/posts/251209_4.png)

사실 그동안 나에게 **바이브 코딩(Vibe Coding)** 은 그리 긍정적인 이미지가 아니었다. 물론 나조차 요즘 AI에 거의 의존하다시피 하고 있지만, 크고 작은 오류를 마주할 때마다 "과연 대학생 입장에서 이렇게 지식을 쌓아가는 게 맞는 걸까?"라는 의구심이 들었기 때문이다. 원리를 모른 채 결과만 내는 것이 성장을 저해하는 것 같아 불안하기도 했다.

하지만 이번 세션을 통해 현업에서는 이것이 선택이 아닌 필수라는 점을 체감했다. 특히 6개월 동안 여러 명이 매달려도 해결하지 못했던 버그를 **Cursor**를 활용해 단 한 시간 만에 해결했다는 사례는 정말 충격적이었다. 원래라면 한 달은 꼬박 걸렸을 프로젝트를 단 3일 만에 끝내는 효율성을 보며, 내가 가졌던 고민이 도구의 활용이라는 측면에서 다시 정립되는 기분이었다.

세션을 듣고 나니, 당장 나부터 실행해 볼 수 있는 것들이 명확해졌다.

- 기획서 대신 UI 뼈대부터: 장황한 기획서를 쓰는 대신 Cursor로 UI를 먼저 뽑아보기. 기획 단계를 줄이면 인생의 속도가 빨라진다.

- AI를 디딤돌 삼아 역할 넘나들기: 기획자도 코드를 만지고, 개발자도 디자인에 관여하며 작은 영역부터 경계를 허물어보기.

- 적극적인 위임: PR 리뷰, 로그 분석, 문서 정리, 단순 반복 업무 등 AI가 더 잘할 수 있는 일은 과감히 맡기기.

이런 식으로 AI를 내 도구로 잘 길들인다면, 나중엔 나 혼자서도 하나의 프로젝트를 온전히 완수할 수 있지 않을까? 라는 생각이 들었다.


### 모두를 위한 친절한 언어모델(LLM) 학습 가이드

전에 RAG 프로젝트를 해보면서 LLM에 대해 관심이 높아져서 선택한 세션이다. 세션 제목만 봐도 알다시피 학습 가이드를 주는 세션이였다.

1. LLM API vs 나만의 LLM (sLLM)
가장 먼저 고민하게 되는 지점은 남이 만든 좋은 모델(API)을 쓸 것인가 아니면 직접 만들 것인가이다.

- **LLM API (GPT-4, Claude 등)**: 성능이 압도적이고 관리가 편하지만, 비용 문제와 보안상의 이유로 기업 내부 데이터를 태우기엔 부담이 있다.

- **나만의 LLM (Open Source, sLLM)**: Llama 3나 Mistral 같은 오픈소스를 가져와 특정 목적에 맞게 튜닝하는 방식이다. 인프라 비용은 들지만 데이터 보안이 철저하고 우리 도메인에 특화된 '똑똑한 비서'를 만들 수 있다는 장점이 있다.

2. 모델이 만들어지는 과정: 데이터가 곧 실력이다.
LLM은 단순히 코딩으로 만드는 게 아니라, 거대한 데이터를 어떻게 잘 먹이느냐의 싸움이었다.

데이터 수집 → 필터링 → 학습 순으로 진행되는데, 특히 필터링 단계에서 데이터의 퀄리티, 유해성, 교육적 가치를 따지는 과정이 모델의 인성을 결정짓는다. 아무리 데이터가 많아도 쓰레기가 들어가면 쓰레기가 나온다는(GIGO) 원칙이 여기서도 적용된다.

3. 학습(Training) vs 추론(Inference)
세션에서 가장 인상 깊었던 LLM의 메커니즘을 정리해 보았다.

[학습: 모델의 뇌를 만드는 과정]
- **Pre-train** : 인터넷의 방대한 데이터를 통째로 읽어 기초 상식을 쌓는 과정.
- **Fine-tuning** : 특정 목적(예: 법률, 의료, 통계)에 맞게 소규모 데이터를 추가 학습시키는 과정.
- **RLHF (사람 피드백 기반 강화학습)**: 사람이 직접 답변의 선호도를 매겨, 모델이 더 '사람답고 친절하게' 대답하도록 교정하는 기술이다. (모델이 아니라 학습 기법 중 !)
- **RLVR (보상 기반 강화학습)**: 수학 문제 풀이나 코드 작성처럼 정답이 명확한 분야에서 성능을 높이기 위해 사용하는 최신 강화학습 기법이다. (이 부분은 생소했는데, 고난도 추론을 가능하게 하는 핵심 기술이었다.)

[추론: 만들어진 뇌를 활용하는 과정]

학습된 모델을 실제로 사용하는 단계다. 개인 환경에서도 **Ollama**같은 도구를 활용하면 내 컴퓨터 안에서 가볍고 빠르게 LLM을 돌려볼 수 있다는 점이 매력적이었다.     

![251209_5.png](/assets/img/posts/251209_5.png)

세션에서 보여준 LLM 필수 지식(?) 사진을 보니, 아직 공부해야 할 용어들이 참 많았다. 특히 RLHF나 RLVR 같은 강화학습 개념은 통계학과 전공 지식과도 맞닿아 있는 부분이 많아 보여, 앞으로 더 깊게 파보고 싶다는 욕심이 생겼다.

막연했던 LLM의 세계가 조금씩 선명해지는 기분이다. RAG 프로젝트에서 느꼈던 한계들을 이런 학습 기법들로 어떻게 해결할 수 있을지 고민하며, 다음 공부 계획을 세워봐야겠다!

### AI 원시인의 바이브 코딩 경험기: 개발자의 미래에 관한 생각


이번 세션은 앞서 들었던 세션의 확장판 같은 느낌이었다. 흥미로운 점은 연사분이 비개발자로서 직접 서비스를 개발하며 겪은 생생한 바이브 코딩의 모든 것을 공유해 주셨다는 것이다.

바이브 코딩의 핵심은 속도와 도파민이다. 문법 규칙에 얽매이지 않고 머릿속의 목표를 AI에게 던지면, 노력 대비 압도적인 결과물이 즉각적으로 튀어나온다.

- 기술 검토: 기술 제안부터 상세 구현 방법까지 AI가 가이드를 잡아준다.
- 뼈대 잡기: 정형화된 패턴 구현에서 AI는 사람보다 훨씬 빠르다.
- 비동기 멀티태스킹: AI에게 일을 시켜놓고 나는 다른 생각을 하는 것, 이것이 진정한 비동기 프로그래밍이 아닐까?

물론 장점만 있는 것은 아니었다. 비개발자가 바이브만으로 MVP(최소 기능 제품)를 넘어설 때는 반드시 한계가 찾아온다. 개발 환경 설정부터 막히거나, MVP 이상의 디테일을 구현하려 할 때 프롬프트는 점점 더 구체적이고 정교해져야만 했다. 결국 돌아가기만 하면 된다는 주의로 속도를 냈지만, 결국 명확한 한계를 넘기 위해선 개발자의 언어와 개입이 필요해지는 순간이 온다.

세션을 통해 얻은 가장 큰 인사이트는 바이브 코딩도 결국 개발 경험이 있는 사람이 훨씬 더 잘한다는 사실이다. AI는 나를 대체하는 것이 아니라, 나라는 팀원을 확장하는 강력한 도구다. 이제 실력은 코드를 치는 속도가 아니라, 얼마나 명확하게 질문하고 프롬프팅하느냐에서 결정된다. 단순 반복은 AI에게 빠르게 맡기고, 우리는 아이디어를 프로덕트로 풀어내는 설계와 기획의 본질에 더 집중해야 한다.

이제는 침대에 누워 태블릿과 목소리만으로 코딩을 하는 입 코딩의 시대가 머지않았다. 통계학을 전공하며 데이터를 다루는 나에게도, 이 도구들을 어떻게 활용해 나만의 퍼즐을 완성할지가 가장 큰 숙제이자 기회가 될 것 같다.

결국 중요한 건 아이디어, 그리고 그것을 AI로 실현시키는 능력이라고 생각한다.

### Google이 AI 알고리즘만 연구할까요?

마지막은 가볍게 들었지만 울림은 컸던 세션이였다. Google이 알고리즘을 넘어 현실의 문제를 어떻게 푸는지 보여주었다.

- **기후 예측 (NeuralGCM)**: 순수 머신러닝의 오류(데이터 드리프트)를 물리 법칙과 결합해 해결한다.
- **의료 혁신 (유방암 검진)**: AI가 의사를 대신하는 게 아니라, 위험 환자를 우선순위로 지정해 의사에게 전달함으로써 골든타임을 확보한다.

결국 AI는 혼자 존재할 때보다 도메인 지식과 결합될 때 더 효율적인 도구로 변한다는 것을 느꼈다. 단순히 모델 성능을 높이는 법을 고민하기보다, 내가 가진 지식으로 세상의 문제에 연결해볼까?를 고민해보는 분석가이자 개발자가 되고 싶다는 생각이 든 시간이었다.
